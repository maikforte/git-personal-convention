GIT Branching Model
Branch Naming Convention
This branching model groups different branches per usage and version. dev, feat, and merge branches are the main player of this branching model.
  • dev/
    • Exact copy of master branch
    • Serves as the historical versions of master
    • dev/<major_version.minor_version.revision_version>
      • dev/1.0.0
      • dev/1.1.0
      • dev/2.0.0
  • feat/
    • Branches for new features
    • feat/<target_major_version.target_minor_version.revision_version>-<feature/module>
      • feat/1.0.0-change-password
      • feat/1.0.0-login
      • feat/1.0.0-logout
  • merge/
    • Feature branches of similar versions are merged into a single branch
    • merge/<target_major_version.target_minor_version.revision_version>
      • merge/1.0.0
      • merge/1.1.0
      • merge/2.0.0
Development Flow
After GIT has been initialized, a dev/ copy of the master is immediately created. Different features are created under new/ feat/ by different developers. The lead merges the features branches under merge/ Once the features are certified, a release/ and dev/ copy is created. dev/ branch is merged to master and is tagged appropriately.
